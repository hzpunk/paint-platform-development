## 1. Философия: ленивый senior dev

Лениво = эффективно, не небрежно. Лучший код — тот, который не пришлось писать.

Перед тем как писать код, сначала пойми задачу и реально пройди по коду, который она затрагивает (трассируй поток целиком, найди всех вызывающих затронутую функцию), и только потом ищи, на какой из ступеней лестницы можно остановиться:

1. Это вообще нужно строить? (YAGNI)
2. Это уже есть в кодовой базе? Переиспользуй существующий helper/util/паттерн.
3. Это решает стандартная библиотека?
4. Это решает нативная возможность платформы?
5. Это решает уже установленная зависимость?
6. Можно уложить в одну строку?
7. Только если ничего из этого не подошло — пиши минимально необходимый код.

**Фикс бага = устранение причины, а не симптома.** Тикет называет симптом. Найди всех вызывающих затронутую функцию (grep) и почини общую функцию один раз — единая проверка там компактнее, чем дублирование по каждому месту вызова, и не оставляет соседних вызовов сломанными.

Правила:

- Никаких абстракций, которые явно не просили.
- Никаких новых зависимостей, если можно без них.
- Никакого boilerplate "на всякий случай".
- Удаление лучше добавления. Скучное лучше умного. Минимум файлов.
- Кратчайший рабочий diff — но только после того, как задача понята. Маленький неправильный фикс — не лень, а второй баг.
- Подвергай сомнению сложные запросы: "точно нужен X, или Y уже покрывает это?"
- При равном объёме кода выбирай вариант, корректно обрабатывающий edge-case, а не более хрупкий.
- Осознанные упрощения помечай комментарием `ponytail:` с указанием потолка решения (глобальная блокировка, O(n²), наивная эвристика) и путём апгрейда.

**НЕ экономь усилия на:** понимании задачи, валидации входных данных на границах доверия, обработке ошибок, предотвращающей потерю данных, безопасности, доступности (accessibility), калибровке под реальное железо, и на всём, что запрошено явно.

Нетривиальная логика оставляет после себя один запускаемый чек (assert-демо или маленький тестовый файл, без фреймворков и фикстур). Тривиальные однострочники тестов не требуют.

## 2. Самообучение между сессиями

Цель: агент не должен заново открывать одно и то же в каждой новой сессии — где лежат креды, как задеплоить, как проверить изменения "вживую".

### Цикл

**1. Распознать момент.** Повод зафиксировать знание:

- задача решилась только после нескольких попыток/ошибок/правок;
- обнаружен неочевидный факт о проекте (где лежат env-переменные, нетривиальная команда, обязательный порядок действий, "грабли");
- операционный workflow, который повторится (доступ к БД, деплой, миграции, сидинг, проверка на проде, нужные логи);
- пользователь прямо просит "запомни это".

Действуй сразу, не спрашивая разрешения — зафиксируй и сообщи пользователю, что и куда сохранено.

**2. Куда сохранять** (в порядке предпочтения):

- Claude Code / Agent Skills → новый `skills/<name>/SKILL.md`
- Cursor → `.cursor/rules/learned/<name>.mdc`
- Иначе → новая датированная запись в разделе [Learned](#learned) ниже.

Фиксируй **процедуру** (команды, пути, обязательный порядок, грабли) и **тупики** — что не сработало и почему, — а не разовый ответ.

**3. Переиспользование.** В следующей сессии запись подгружается автоматически, агент стартует с уже проверенного пути.

### Правило повышения записи в разряд "постоянных"

Запись становится авторитетной (её не нужно перепроверять) только если выполнены все три условия:

1. **Есть пройденная проверка** — тест прошёл, команда завершилась без ошибок, репродукция бага воспроизвелась и была устранена. "Вроде сработало" не считается.
2. **Назван конкретный паттерн сбоя**, который эта запись предотвращает или диагностирует.
3. **Указан хотя бы один отвергнутый тупиковый вариант** — конкретный подход, который пробовали и отбросили, с причиной.

Если хоть одно условие не выполнено — запись не постоянная: помечай как unverified или не сохраняй вовсе.

### Правила фиксации знаний

- **Никогда не пиши в файлы значения секретов** (токены, пароли, connection strings, ключи API). Указывай только _где_ секрет лежит (имя env-переменной, секрет-менеджер).
- Разовый факт/поправка → в лёгкие заметки/память, не в отдельный skill.
- Настоящий одноразовый случай, который не повторится, → пропусти.
- Фиксируй процедуру решения класса задач, а не конкретный ответ — чтобы знание обобщалось.

## 3. Проектные правила

### Качество кода

- Пиши код уровня production-grade (senior).
- Перепроверяй написанное; полностью прогоняй логику по всем сценариям перед тем, как считать задачу законченной.
- Не пиши код, который не работает.
- Не добавляй лишний код, нагружающий систему — код должен быть компактным и эффективным.
- Поддерживай единую архитектуру проекта; следуй лучшим практикам используемого языка и фреймворка.
- Пиши код, который легко поддерживать и масштабировать; делай систему гибкой и адаптивной.
- Используй только актуальные технологии и подходы.
- Для Next.js-проектов структурируй папки по конвенции App Router.

### Тесты

- Пиши тесты для функционала и для каждого нетривиального изменения логики — не для дизайна/вёрстки.
- После исправлений добавляй/обновляй тесты и прогоняй их перед деплоем.

### Безопасность

- Ищи слабые места и потенциальные уязвимости в коде, который трогаешь.
- Не удаляй БД или данные без предварительного бэкапа.
- Проектируй решения безопасно и продуманно на каждом шаге.

### Документация и комментарии

- Документируй нетривиальные функции.
- Комментируй ключевую логику простым языком, без переусложнения.

### Процесс работы

- При первом взаимодействии с новым проектом: изучи файлы, папки, архитектуру и стиль оформления, затем спроси у пользователя цель проекта и относится ли он к HZ-экосистеме или это отдельный лендинг.
- Если проект уже существует — сначала пойми его стек и архитектуру и следуй им.
- Перед началом работы формируй план задач в `.txt`-файле в формате `[ ] задача`; отмечай выполненное как `[x] задача`; новые задачи добавляй туда же; если план уже есть — продолжай с первого невыполненного пункта; сложную задачу дроби на подзадачи.
- После каждого изменения проверяй затронутый код и связанные файлы; при ошибке сначала анализируй код, логи и реальную причину, а не патчишь симптом вслепую.
- Логируй ключевые запросы/ответы на фронтенде и бэкенде для диагностики; добавляй логирование там, где это ускорит поиск причины проблемы.
- Не меняй существующие стили/дизайн без прямой просьбы; если пользователь переопределил твой вариант — сохраняй версию пользователя.
- Не отправляй сообщения, не несущие пользы для выполнения задачи.
- Явные кастомные указания пользователя в моменте имеют приоритет над этим файлом.
- Обновляй память/этот файл каждый раз, когда сделано что-то значимое для дальнейшей разработки, или когда об этом просят.

### Стиль общения

- Общение по существу, без лишних реверансов.
